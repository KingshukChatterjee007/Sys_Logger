2. DISTRIBUTED ARCHITECTURE ("Why simple servers fail")
=============================================================================
When scaling to 1,000+ units, a simple app crashes because:
1. RAM: Storing 1,000 units in a Python dictionary (`units = {}`) fills memory.
2. CPU: Looping through 1,000 units to calculate averages freezes the server.
3. Disk: Writing logs to C:\Usage_Logs doesn't work across multiple servers.

THE SOLUTION:
1. REDIS (The Fast Brain):
   - Replaces `units = {}`.
   - Shared "live" state across all 10 servers.
   - Handles real-time Pub/Sub for the Dashboard.

2. POSTGRESQL (The Deep Memory):
   - Replaces local .log files.
   - Stores historical data permanently.
   - Uses "Partitioning" (Splitting tables by month) to stay fast.

3. WORKERS (The Muscle):
   - Background processes (Celery) that do the heavy math so the API stays fast.

=============================================================================
3. DATA FLOW: "THE JOURNEY OF A METRIC"
=============================================================================
How we handle the firehose of data without crashing:

1. INGESTION (The "Catch"):
   - 1,000 Units send data to the API.
   - API Server does NOT write to disk. 
   - API Server dumps data into Redis (Memory) -> 5ms response time.

2. PROCESSING (The "Filter"):
   - Worker Server wakes up.
   - Pulls 100 records from Redis at once.
   - Checks Alerts (CPU > 90%).
   - Batch Inserts 100 records into Postgres (1 write query instead of 100).

3. VISUALIZATION (The "View"):
   - Redis "broadcasts" the update to all connected Dashboards instantly.
   - Dashboard queries Postgres only when asking for long-term history.
